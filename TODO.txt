## BUG — AI does not play the J (Jack / Swap) card
**File:** `backend/src/game/ai-strategy.ts`

The AI currently discards the J card instead of playing it. The J card allows
swapping one of the AI's marbles with an opponent's marble.

Steps to implement:
- In `ai-strategy.ts`, add a pass that evaluates J card actions.
- For each opponent marble (not on a START_POSITION of its own color, not in
  ARRIVAL_POSITIONS, not in HOME_POSITIONS), check if swapping with one of the
  AI's own eligible marbles constitutes a legal move using `getLegalAction()` in
  `move-validator.ts`.
- Prioritize swaps that benefit the AI (e.g., swap a marble close to winning with
  an opponent far away, or rescue an AI marble that is exposed).
- A basic non-strategic implementation is acceptable as a first step: pick any
  legal swap target randomly rather than discarding the card.

## BUG — Swap (J card) incorrectly restricted on other players' START_POSITIONS
**File:** `packages/shared/src/move-validator.ts`

Currently, a marble cannot be swapped if it sits on ANY `START_POSITION` on the
board. This is too restrictive. The rule should be:

> A marble CANNOT be the target of a swap if and only if it is on its OWN
> player's START_POSITION (i.e., the start position matching its color).
> Being on another player's start position must NOT prevent it from being swapped.

Steps to fix:
- Locate the swap validation logic in `move-validator.ts` (the check that
  prevents swapping marbles on start positions).
- Replace the blanket `isStartPosition()` check with a per-color check:
  only reject the target marble if `getStartPosition(targetMarble.color) ===
  targetMarble.position`.
- Confirm with `board-config.ts` which helper function or constant lists each
  player's specific start position.
- Add test cases: marble on own start → swap rejected; marble on another
  player's start → swap allowed.

The current animation for this combined "enter + capture" action looks broken.

- Currently the enter animation and the capture animation may be running
  simultaneously or in the wrong order.
- Fix the sequence:
  1. Show the enemy marble disappearing from the start position (captured).
  2. Then animate the new marble appearing/entering on the start position.
- Review animation duration constants in `packages/shared/src/constants.ts`
  (e.g., `ENTER_DURATION`, `CAPTURE_DURATION`) to ensure timing feels correct
  and consistent with other animations.

## BUG — Promotion move incorrectly rejected as illegal for the last marble entering ARRIVAL
**File:** `packages/shared/src/move-validator.ts`

Reproduction steps:
1. Play as red.
2. Have only one marble remaining outside ARRIVAL_POSITIONS.
3. That marble is positioned such that the exact remaining distance to the
   last ARRIVAL slot requires a specific card (e.g., marble at main-path
   position 7 needs a 2 card to land on ARRIVAL position index 38).
4. The player holds that exact card.
5. Expected: The move is valid and playable.
6. Actual: The move is rejected as illegal, all cards are discarded.

Root cause (suspected):
- `getLegalAction()` or the promote path calculation in `move-validator.ts`
  may not correctly compute the remaining distance when transitioning from the
  main path into the ARRIVAL zone.
- The promotion logic must account for the fact that once a marble crosses its
  own START_POSITION, subsequent steps go into ARRIVAL_POSITIONS (indexed
  separately from MAIN_PATH).
- Check the boundary condition when a marble is already on the main path near
  the start position and a small card value would land it exactly on the last
  ARRIVAL slot.

Steps to fix:
- Trace `getLegalAction()` for a promote scenario to find where the distance
  calculation fails.
- Ensure the promote path correctly maps remaining steps onto ARRIVAL_POSITIONS
  indices without off-by-one errors.
- Verify the fix works for all four player colors (each has their own
  START_POSITION and ARRIVAL_POSITIONS layout in `board-config.ts`).
