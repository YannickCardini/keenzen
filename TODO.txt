BUG 1 — Sign-in error persists after backend recovery (requires page refresh)
--------------------------------------------------------------------------------

Context:
  - Auth flow uses a Google OAuth redirect: `AuthService.login()` calls
    `GoogleSignIn.signIn()`, which redirects the page to Google, then Google
    redirects back with an `id_token` in `window.location.hash`.
  - The `AuthService` constructor detects the hash and calls
    `handleRedirectCallback()`, which POSTs the idToken to `POST /api/auth/google`.
  - If the backend is down, the POST fails, `loginError$.next('server')` is
    emitted, and the error banner appears for 4 seconds.
  - When the user retries, `GoogleSignIn.signIn()` is called again. However, the
    stale `id_token` is still present in `window.location.hash`. The Capacitor
    plugin may short-circuit and reuse the old (possibly expired) token instead
    of starting a fresh OAuth redirect, causing the POST to fail again.
  - A full page refresh clears the hash, which is why it fixes the issue.

Files to modify:
  - `frontend/src/app/services/auth.service.ts`

Steps:
  1. At the END of `handleRedirectCallback()` (in both the `try` and `catch`
     branches, or in a `finally` block), clear the URL hash so the stale
     `id_token` is removed:
       history.replaceState(null, '', window.location.pathname + window.location.search);
  2. This ensures that when the user retries, `GoogleSignIn.signIn()` initiates
     a fresh OAuth redirect rather than processing a stale token from the URL.

BUG 2 — Player info panel shows "Guest player / Not signed in" for authenticated users
--------------------------------------------------------------------------------

Context:
  - When hovering or clicking on a player's avatar in the board, the profile
    panel (`board.component.ts:openProfilePanel`) checks `player?.userId` to
    determine whether the player is a guest (`isGuest = !player?.userId`).
  - If `userId` is missing from the `Player` object in the game state, the panel
    always shows "Guest player — Not signed in".
  - The root cause: `sendJoinMatchmaking()` in `game-state.service.ts:411` does
    not include `userId` in the WebSocket message, even though the
    `JoinMatchmakingMessage` type (defined in `packages/shared/src/types.ts:211`)
    has a `userId?: string` field.
  - In `home.page.ts:113-115`, `sendJoinMatchmaking(playerName, playerPicture)`
    is called but the authenticated user's `id` (`user?.id`) is never passed.

Files to modify:
  1. `frontend/src/app/game/services/game-state.service.ts`
     - Add a `userId?: string` parameter to `sendJoinMatchmaking(playerName?, picture?, userId?)`.
     - Include `userId` in the JSON message sent to the backend:
         { type: 'joinMatchmaking', playerName, browserId, picture, userId }

  2. `frontend/src/app/home/home.page.ts`
     - In `openMatchmaking()`, retrieve `user?.id` alongside `user?.name` and
       `user?.picture`.
     - Pass it as the third argument:
         this.gameStateService.sendJoinMatchmaking(playerName, playerPicture, user?.id);

  Note: Verify that the backend's matchmaking handler (`session-manager.ts` or
  `matchmaking-manager.ts`) correctly reads `userId` from `JoinMatchmakingMessage`
  and passes it into the `PlayerConfig`, so it ends up in the `Player` object
  broadcast in the game state.

FEATURE 3 — Profile editing in the login modal (name + profile picture)
--------------------------------------------------------------------------------

Context:
  - When a user is signed in, the login modal (`home.page.html:135-143`) shows
    their avatar, name, stats, and a logout button — but no way to edit their
    profile.
  - The `UserDoc` in Cosmos DB (managed by `backend/src/auth/auth-router.ts`)
    stores `name` and `picture` fields that currently only come from Google at
    first login.
  - A new `PATCH /api/auth/user/:id` backend endpoint and new frontend editing
    UI are required.

Sub-tasks:

  A. Backend — Add PATCH /api/auth/user/:id endpoint
     File: `backend/src/auth/auth-router.ts`

     - Add a `PATCH /api/auth/user/:id` route.
     - The request body accepts: `{ name?: string, picture?: string }`.
     - Validate `name`:
         - Must be between 1 and 30 characters.
         - Allowed characters: letters (unicode), digits, spaces, hyphens,
           underscores, apostrophes. Reject anything else (especially `<`, `>`,
           `"`, `&`, backticks, etc.) to prevent XSS/injection.
     - Validate `picture` (if provided):
         - Accept only a base64-encoded data URL with MIME type `image/jpeg`,
           `image/png`, or `image/webp`.
         - Reject data URLs that exceed 2 MB of raw base64 content.
         - Reject any non-data-URL value (external URLs are not accepted).
     - If validation passes, update the `name` and/or `picture` fields of the
       `UserDoc` in Cosmos DB (use `container.item(id, id).replace(...)`).
     - Return the updated user object (same shape as `POST /api/auth/google`
       response: `{ id, email, name, picture, points, ranking, createdAt }`).

  B. Frontend — Add update method to AuthService
     File: `frontend/src/app/services/auth.service.ts`

     - Add a method `updateProfile(name: string, picture: string): Promise<void>`.
     - It sends `PATCH ${environment.apiUrl}/api/auth/user/${user.id}` with
       `{ name, picture }`.
     - On success, update `user$.next(updatedUser)` and overwrite
       `localStorage` (same key `auth_user`).
     - Expose an `updateError$` Subject<string> to surface validation errors.

  C. Frontend — Add editing UI in the login modal
     Files: `frontend/src/app/home/home.page.html`,
             `frontend/src/app/home/home.page.ts`

     - In the authenticated section of the modal (the `*ngIf="auth.user$ | async as user"` block),
       add an "Edit profile" toggle button next to the logout button.
     - When toggled, show an edit form containing:
         1. A text input for the display name (pre-filled with `user.name`),
            with a maxlength of 30 and a pattern or validation feedback.
         2. A file input for the profile picture, accepting only
            `image/jpeg,image/png,image/webp` and limited to 2 MB client-side.
            Show a preview of the selected image before saving.
         3. A "Save" button (disabled while saving).
         4. An error message area if validation or the API call fails.
     - In `home.page.ts`:
         - When the user selects a file, read it with `FileReader.readAsDataURL()`,
           validate the MIME type and file size client-side before sending.
         - Reject images where either dimension exceeds 512 px (use an
           `<img>` drawn onto an `OffscreenCanvas` to check dimensions).
         - On "Save", call `auth.updateProfile(name, base64DataUrl)` and close
           the edit form on success.
