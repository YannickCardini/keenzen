## 1. Loading indicator on the "Create new room" button (Custom Game modal)

**Where:** Custom Game modal, "choose" stage.
- Template: [frontend/src/app/home/home.page.html:376-379](frontend/src/app/home/home.page.html#L376-L379) (`button.custom-create-btn` calling `createCustomRoom()`).
- Component method: [frontend/src/app/home/home.page.ts:444-453](frontend/src/app/home/home.page.ts#L444-L453) (`createCustomRoom()`).
- Stage transition is driven by `customRoomStatus$` subscription at [frontend/src/app/home/home.page.ts:396-407](frontend/src/app/home/home.page.ts#L396-L407), which sets `customStage = 'in-room'` only after the server replies.

**Problem:**
In production, there is a noticeable delay between clicking "Create new room" and the modal switching to the "in-room" stage. The delay is caused by the websocket round-trip: `gameStateService.connect(...)` opens a new WS connection, then `sendCreateCustomRoom(...)` is sent, and the UI only updates when the backend emits the `customRoomStatus` event. During this window, the button gives no visual feedback, so the user may think the click was ignored and click again.

**Goal:**
Show a loading state on the "Create new room" button (and prevent re-clicks) from the moment it is clicked until either:
- the modal transitions to `customStage === 'in-room'`, or
- an error occurs (`customError` set, or `connectionError$` / `actionRejected$` fires).

**Suggested implementation steps:**
1. Add a new comthis.gameStateService.connectponent flag, e.g. `customCreating = false`, alongside the existing `customStarting` flag at [frontend/src/app/home/home.page.ts:109](frontend/src/app/home/home.page.ts#L109). Reset it to `false` in `resetCustomGameState()` ([home.page.ts:378-391](frontend/src/app/home/home.page.ts#L378-L391)).
2. In `createCustomRoom()` ([home.page.ts:444-453](frontend/src/app/home/home.page.ts#L444-L453)):
   - Guard early-return if `customCreating` is already `true`.
   - Set `customCreating = true` before calling `gameStateService.connect(...)`.
   - Clear `customCreating = false` once the stage transitions to `'in-room'` (inside the `customRoomStatus$` subscription at [home.page.ts:396-407](frontend/src/app/home/home.page.ts#L396-L407)) and inside the error/rejection handlers at [home.page.ts:416-435](frontend/src/app/home/home.page.ts#L416-L435).
3. In the template ([home.page.html:376-379](frontend/src/app/home/home.page.html#L376-L379)):
   - Add `[disabled]="customCreating"` to the button.
   - Swap the label/icon for a loading state when `customCreating` is true (mirror the existing `customStarting ? 'Starting…' : 'Start'` pattern at [home.page.html:511-513](frontend/src/app/home/home.page.html#L511-L513) — e.g. `{{ customCreating ? 'Creating…' : 'Create new room' }}`, optionally with a spinner replacing the `ab-marble` element).
4. Apply the same treatment to the "Join" button at [home.page.html:395-397](frontend/src/app/home/home.page.html#L395-L397) / `joinCustomRoomByCode()` ([home.page.ts:455](frontend/src/app/home/home.page.ts#L455)) if it shares the same delay (verify by inspecting the same WS round-trip).

**Acceptance criteria:**
- Clicking "Create new room" immediately disables the button and shows a loading affordance (text change and/or spinner).
- The loading affordance disappears as soon as the modal switches to the "in-room" stage, or when an error is shown.
- A second click during the loading window is a no-op (no duplicate WS connections, no duplicate `createCustomRoom` messages sent).
