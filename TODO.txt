# TODO

## 1. Accept profile pictures of any input size by resizing/compressing client-side

**Context**
- Profile picture upload is handled in [frontend/src/app/home/home.page.ts](frontend/src/app/home/home.page.ts#L103-L135) (`onFileSelected`).
- The selected file is read into a base64 data URL and sent via `auth.updateProfile(...)` ([frontend/src/app/services/auth.service.ts:101-109](frontend/src/app/services/auth.service.ts#L101-L109)) to `PATCH /api/auth/user/:id`.
- The backend validator in [backend/src/auth/auth-router.ts:114-130](backend/src/auth/auth-router.ts#L114-L130) only accepts:
  - MIME type `image/jpeg`, `image/png`, or `image/webp`
  - Base64 data URL whose decoded payload is ≤ 2 MB
- The frontend currently *rejects* any input that does not already meet the server constraints:
  - Wrong MIME type → "Only JPEG, PNG, or WebP images are allowed."
  - File size > 2 MB → "Image must be under 2 MB."
  - Natural dimensions > 512×512 px → "Image dimensions must not exceed 512×512 px."

**Goal**
Stop rejecting oversized or differently-formatted images. Instead, transparently transform any user-selected image so that the data URL sent to the backend always satisfies the existing server-side constraints. The backend validator must remain unchanged.

**Target output constraints (must be enforced after transformation)**
- Format: `image/jpeg`, `image/png`, or `image/webp` (pick one canonical output format — `image/webp` recommended for best size; fall back to `image/jpeg` if WebP encoding is unavailable).
- Max dimensions: 512×512 px (preserve aspect ratio; downscale only, never upscale).
- Max payload size after base64 encoding: ≤ 2 MB (target a comfortable margin, e.g. ≤ 1.5 MB).

**Sub-tasks**

1. **Broaden the file input acceptance**
   - In [frontend/src/app/home/home.page.html:150](frontend/src/app/home/home.page.html#L150), change `accept="image/jpeg,image/png,image/webp"` to `accept="image/*"` so the OS picker allows any image type (HEIC, GIF, BMP, TIFF, etc., subject to browser support).

2. **Remove the hard rejections in `onFileSelected`** ([home.page.ts:108-116, 122-126](frontend/src/app/home/home.page.ts#L108-L126))
   - Drop the MIME-type whitelist check.
   - Drop the 2 MB pre-check.
   - Drop the 512×512 dimension check.
   - Keep a sanity check that the selected `File` is actually decodable as an image (i.e. handle `img.onerror` by surfacing a friendly "Could not read image." error).

3. **Implement a client-side image transformer**
   - Create a helper (e.g. `frontend/src/app/services/image-utils.ts` or a private method in `HomePage`) with a signature like:
     `async function normalizeProfileImage(file: File): Promise<string>` returning a base64 data URL that satisfies the target constraints.
   - Steps:
     a. Read the file into an `HTMLImageElement` (via `URL.createObjectURL` or `FileReader.readAsDataURL`).
     b. Compute target dimensions: scale down so the longer side ≤ 512 px, preserving aspect ratio. Do not upscale.
     c. Draw onto an offscreen `<canvas>` of the target dimensions.
     d. Export with `canvas.toDataURL('image/webp', quality)` (fallback to `'image/jpeg'`).
     e. If the resulting data URL exceeds the 2 MB (or chosen ≤ 1.5 MB) budget, iteratively reduce `quality` (e.g. start at 0.9, step down by 0.1, floor at 0.4) and re-export until it fits.
     f. If the budget still cannot be met at the lowest quality, additionally reduce dimensions (e.g. 384, then 256) and retry.
   - Release any object URLs created (`URL.revokeObjectURL`) once the image is loaded.

4. **Wire the transformer into the upload flow**
   - Replace the current logic in `onFileSelected` so it calls `normalizeProfileImage(file)` and assigns the resulting data URL to both `this.editPreviewUrl` and `this.editPictureDataUrl`.
   - Show a non-blocking status while the transformation runs (the canvas work is fast but a very large source image may take a moment).
   - On transformer failure, set `this.editError` to a clear message (e.g. "Could not process image, please try another file.").

5. **Edge cases to handle explicitly**
   - Animated formats (GIF, animated WebP): only the first frame needs to be kept — the canvas pipeline naturally handles this, but verify.
   - Formats the browser cannot decode natively (e.g. HEIC on non-Safari browsers): the `<img>` load will fail; surface the "Could not read image." error instead of crashing.
   - EXIF orientation: if the source is a JPEG from a phone camera, ensure the rotation is respected (use `createImageBitmap(file, { imageOrientation: 'from-image' })` when available, otherwise document this as a known limitation).
   - Transparent PNGs exported to JPEG: prefer WebP to preserve alpha; if falling back to JPEG, fill the canvas with a solid background (e.g. white) before drawing.

6. **Verification**
   - Manually test with: a 4000×3000 px JPEG > 5 MB, a 1024×1024 px PNG with transparency, a small 200×200 px WebP (must not be upscaled), and a non-image file (should show error).
   - Confirm the request payload sent to `PATCH /api/auth/user/:id` always passes the server-side validator in [backend/src/auth/auth-router.ts:114-130](backend/src/auth/auth-router.ts#L114-L130).
