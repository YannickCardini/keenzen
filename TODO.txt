## Bug 1 — Incorrect preview destination when card "7" results in a promotion

**Files involved:**
- `frontend/src/app/home/components/board/board.component.ts` — `previewInfo` computed signal (lines 107–119)
- `packages/shared/src/move-validator.ts` — `buildMoveAction` (currently unexported)

**Description:**
When the player selects the "7" card and has already chosen the step count for the first marble (i.e., `sevenFirstSteps < 7`),
the preview path destination is computed by calling `getPositionAfterMove(focusedMarble, stepsForFocused)`,
which always returns a position along `MAIN_PATH`. If the move would actually result in a **promotion**
(the marble crosses its own start position and enters the arrival zone),
`getPositionAfterMove` returns the wrong square — the raw MAIN_PATH position after the start, instead of the correct arrival zone square.

For all other cards (non-7), `getLegalAction()` is called, which internally calls `buildMoveAction()` and correctly detects the promote condition,
returning an action of type `'promote'` with `to` pointing to the arrival zone. The 7-split preview branch bypasses this logic entirely.

**Steps to fix:**

1. **Export a new helper function** from `packages/shared/src/move-validator.ts`:
   - Create and export a function `getActionForSteps(card: Card, from: number, steps: number, ctx: LegalMoveContext): Action | null` that is a thin wrapper calling `buildMoveAction(card, from, steps, ctx)` (currently private).
   - This function returns the correct `Action` including `type: 'promote'` and the actual arrival zone square as `to` when applicable.

2. **Update the preview logic** in `board.component.ts`, `previewInfo` computed signal, in the `card.value === '7'` branch where `steps1 < 7 && marble1 !== null`:
   - Replace the call to `getPositionAfterMove(focusedMarble, stepsForFocused)` and the manual construction of a `'move'` action with a call to the new `getActionForSteps` function.
   - Pass a dummy card (e.g., `{ id: '__preview__', value: '7', suit: '♠' }`), the focused marble position, the computed `stepsForFocused`, and the current context `ctx`.
   - Use the returned action (which may be a `'move'` or `'promote'`) in the existing `computePreviewFromAction(action)` call.

## Bug 2 — Marble on its own start position can be illegally captured by a backward move

**Files involved:**
- `packages/shared/src/move-validator.ts` — `buildBackwardMoveAction` (lines 204–234) and `pathIsClear` (lines 256–287)

**Description:**
A marble standing on its **own** start position (i.e., the marble's color matches the start position's color) is supposed to be **invincible**
— it cannot be captured or passed through by other players. This rule is already correctly enforced for **forward** moves:
`pathIsClear()` checks every intermediate square and the destination, and returns `false` if a marble is resting on its own start position.

However, `buildBackwardMoveAction()` (used when card "4" is played, moving backward 4 squares) **does not perform any path checks**. It only checks whether the destination `to` belongs to the current player's own marbles. It does not check:
- Whether the destination `to` is another player's start position with that player's marble on it (should be uncapturable).
- Whether any **intermediate** square along the backward path is a start position with the owner's marble on it (should block passage).

Note: a marble of color X standing on the start position of color Y (a different color)
 has no special protection and can be captured normally — only a marble on its **own** color's start is protected.

**Steps to fix:**

1. **Add a destination check** in `buildBackwardMoveAction`, immediately after `if (ownMarbles.includes(to)) return null;`:
   - If `to` is any player's start position (use `getStartPositionOwner(to)`), and that player's marble is currently at `to` (check `ctx.marblesByColor[owner].includes(to)`), then `return null` — the marble is invincible and the move is illegal.

2. **Add intermediate square checks** in `buildBackwardMoveAction`:
   - Before computing `to`, iterate over the intermediate squares (steps 1 to `steps - 1`) going backward from `from` using `getPositionAfterBackwardMove`.
   - For each intermediate square, apply the same check: if it is a start position and the owning player's marble is there, `return null` — the path is blocked.
   - Alternatively, extract this logic into a new `pathIsClearBackward(from, steps, playerColor, allMarbles, marblesByColor)` helper function (mirroring the existing `pathIsClear` for forward moves) and call it from `buildBackwardMoveAction`.
