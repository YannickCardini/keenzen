TASK 1 — Public Player Profile Page
================================================================================
Goal: Create a new page that displays the public profile of any player.

Routing & access points:
  - New route: `/profile/:id` (add it in `frontend/src/app/app.routes.ts`).
  - Entry point A — Leaderboard: in `frontend/src/app/leaderboard/leaderboard.page.html`,
    each `<li class="lb-row">` must become clickable and navigate to
    `/profile/<userId>`. The leaderboard endpoint currently does not return the

    extend the `LeaderboardEntry` interface in `leaderboard.page.ts` accordingly.
  - Entry point B — In-game: in
    `frontend/src/app/game/components/board/board.component.html` (around
    line 174), the "View leaderboard" button inside the `profile-panel` must be
    replaced with a "View profile" button that opens
    `/profile/<userId>` in a NEW BROWSER TAB (use `window.open(url, '_blank')`).
    The userId is the one already loaded by the panel (`panel.data` /
    `loadProfile` logic in `board.component.ts`).

Page contents (display only — no editing):
  - Profile picture, displayed at its real/native size (do NOT crop or scale
    down; only constrain with a sensible max-width so the layout stays usable
    on small screens).
  - Player name.
  - Total points.
  - Current ranking (rank number).
  - "Member since" date — derived from `createdAt`.
  - "Last seen" date — derived from `lastLogin`. NOTE: `lastLogin` is NOT yet
    exposed by `GET /api/auth/user/:id`; add it to that endpoint's response in
    `backend/src/auth/auth-router.ts`.
  - A "Send private message" button (see sub-task below).

Private messaging (keep it as simple as possible):
  - Backend:
      * Create a new Cosmos DB container `messages` (mirror the pattern used
        for `getUsersContainer` in `backend/src/db.ts`).
      * Add endpoints in a new router `backend/src/messages/messages-router.ts`:
          - `POST /api/messages` — body `{ toUserId, text }`, requires the
            sender to be authenticated (reuse the existing Google id-token
            validation logic from `auth-router.ts`; factor it into a helper if
            needed). Reject if `text` is empty or longer than e.g. 500 chars.
          - `GET /api/messages/inbox` — returns the authenticated user's
            received messages, newest first.
      * Wire the router in `backend/src/index.ts`.
  - Frontend:
      * On the profile page, a "Send message" button opens a small modal with
        a textarea + send button. Send via the new POST endpoint. Show a
        success toast / inline confirmation; show a generic error otherwise.
      * No inbox UI is required for this task (the inbox is handled in
        Task 2); only the send flow needs to work end-to-end here.
  - Guests (not signed in) should see the "Send message" button disabled with
    a tooltip "Sign in to send a message".

Acceptance:
  - Clicking a row on the leaderboard opens that player's profile page.
  - Clicking "View profile" inside the in-game profile panel opens the
    profile in a new tab.
  - Profile page renders all listed fields and lets a signed-in user send a
    message that is persisted in Cosmos DB.

TASK 2 — Inbox (boîte de réception)
================================================================================
Depends on: Task 1 (the `messages` container and `GET /api/messages/inbox`
endpoint must already exist).

Goal: Give signed-in users a way to read the private messages they have
received via the messaging system introduced in Task 1.

Implementation is intentionally left open. Any approach is acceptable as long
as the result satisfies these three criteria:
  1. Simple — minimal UI, no over-engineering, no features beyond reading
     received messages (and ideally seeing which are unread).
  2. Ergonomic — the inbox must be easy to find and easy to use; opening it,
     reading a message, and going back should feel obvious.
  3. Harmonious — visual style, spacing, colors, typography, and animations
     must match the rest of the app (re-use existing components / SCSS
     patterns from the home page, leaderboard, and modals rather than
     introducing new visual idioms).

Suggested directions (non-binding — pick whatever fits best):
  - A dedicated `/inbox` route, or a modal opened from the user menu / avatar.
  - An unread-count badge somewhere visible (e.g. on the avatar).
  - A list of messages with sender name, sender avatar, date, and body.
  - Mark-as-read behaviour can be implicit (on open) or explicit; whichever
    is simpler is fine for v1.

Backend additions are allowed if needed (e.g. an `unreadCount` field, or a
`PATCH /api/messages/:id/read` endpoint), but only add what the chosen UX
actually requires. Guests have no inbox.

Acceptance:
  - A signed-in user who has received messages can open the inbox and read
    them, sorted newest first.
  - The inbox visually fits in with the rest of the app.
  - Guests cannot access the inbox.

TASK 3 — Custom Game (currently shown as "Coming soon")
================================================================================
Goal: Implement the Custom Game flow. The button already exists in
`frontend/src/app/home/home.page.html` (around line 87) but is disabled.

Sub-task 3.1 — Frontend UI
  - Remove `disabled` and `title="Coming soon"` from the Custom Game button.
    Bind `(click)` to a new `openCustomGame()` method in
    `frontend/src/app/home/home.page.ts`.
  - Create a Custom Game modal (similar visual style to the existing matchmaking
    modal, see lines 312–366 of `home.page.html`) containing:
      a) A unique 6-character alphanumeric "room code" (generated by the
         backend on creation), prominently displayed with a copy-to-clipboard
         button. Guests can use this code to join WITHOUT an account.
      b) A list of currently joined players (up to 4), styled like the
         matchmaking slots.
      c) A "Join with code" input INSIDE this same modal (not on the home
         page). After clicking the Custom Game button, the user can either
         create a new room OR paste an existing code to join one. Joining
         via code does NOT require authentication; reuse the same guest-name
         flow as matchmaking (`generateGuestName` in
         `frontend/src/app/shared/guest-name.ts`).
      d) An invite section showing all signed-up players from the DB:
            - Reuse the leaderboard list visual (`leaderboard.page.html`).
            - Sorted by `points` DESC.
            - With a search bar that filters by `name` (case-insensitive,
              client-side filtering is acceptable).
            - Each row has an "Invite" button. Clicking "Invite" simply
              sends a PRIVATE MESSAGE to that player (reusing the messaging
              system from Task 1 — `POST /api/messages`). The message body
              is in ENGLISH and reads roughly:
                "Hi! Would you be interested in joining my custom game?
                 Enter the following code to join: <ROOM_CODE>"
              For v1 we keep it simple — no auto-join, no deep link. A
              future iteration may embed a direct join link in this
              message; do NOT implement that now.
      e) A "Start game" button visible only to the room creator.

Sub-task 3.2 — Backend
  - Create a `CustomGameManager` in `backend/src/session/` modelled on
    `MatchmakingManager` (`backend/src/session/matchmaking-manager.ts`) but
    keyed by a room code instead of a single global queue:
      * `createRoom(creatorWs, creatorInfo)` → returns `{ code }`.
      * `joinRoom(code, ws, playerInfo)` → assigns next free color
        (red → green → blue → orange).
      * `startRoom(code, requesterWs)` → only the creator can call this.
        - If the room has 4 players: start the game immediately (same flow
          as `MatchmakingManager` — instantiate `Game`, register in
          `GameRegistry`).
        - If the room has fewer than 4 players: forward the existing players
          into the public matchmaking queue. They should be added to the
          current `MatchmakingManager` session if one is pending; otherwise a
          new pending session must be created. Preserve their colors when
          possible; reassign if there's a conflict.
  - Wire WebSocket message types (extend the existing protocol used by
    `game-state.service.ts`):
      * `createCustomRoom`, `joinCustomRoom`, `startCustomRoom`,
        `customRoomStatus` (broadcast like `matchmakingStatus`).

Sub-task 3.3 — Edge cases
  - Reject duplicate tab joins from the same browser (mirror the `browserId`
    check in `MatchmakingManager.joinQueue`).
  - If the creator disconnects before starting, the room is destroyed and
    remaining players are notified.
  - Codes expire after e.g. 30 minutes of inactivity.

Acceptance:
  - A signed-in user can open the Custom Game modal, see a generated code,
    and copy it.
  - From the same modal, another user (signed in OR guest) can paste a code
    and join the room.
  - Inviting a DB user from the search list delivers an English private
    message containing the room code, visible in that user's inbox
    (Task 1 + Task 2 dependency).
  - Pressing "Start" with 4 players begins a normal game.
  - Pressing "Start" with <4 players merges everyone into public matchmaking.
