TASK 1 — Declare the last connected player winner when all others disconnect
--------------------------------------------------------------------------------

CONTEXT:
  When a player's WebSocket closes and does not reconnect within 180 seconds,
  `game-messenger.ts` calls `onPermanentDisconnect(color)`, which in turn calls
  `game.ts:markDisconnected(color)`. That method sets `player.isConnected = false`
  and calls `checkAbort()`.

  Currently, `checkAbort()` (game.ts) counts connected humans and only aborts the
  game (reason: 'abandoned') if that count reaches 0. There is no special handling
  for the case where exactly 1 human player remains connected.

WHAT TO DO:
  In `backend/src/game/game.ts`, update `checkAbort()` so that when the number of
  connected human players drops to 1, the game ends with a win for that player
  instead of continuing or aborting as 'abandoned'.

  Steps:
  1. After computing `connectedHumans` in `checkAbort()`, add a branch:
     if `connectedHumans === 1` AND total human players > 1 (i.e. at least one
     has permanently disconnected), find the single remaining connected human
     player and declare them the winner.

  2. Reuse the existing win-end flow used at the bottom of the game loop:
     - Call `applyEndGamePoints(winner.color)` so scoring is applied correctly
       (winner: +3 pts; already-penalized disconnected players keep their -2 and
       are skipped by the scoring guard on `penalizedUserIds`).
     - Broadcast `gameEnded` with `reason: 'win'` and the winner's color.
     - Set `gameFinished = true` and `aborted = true` to exit the game loop.
     - Remove the game from the registry.

  3. Do NOT trigger this path if only bots remain (i.e. the single remaining
     "human" must have `isHuman === true && isConnected === true`).

  4. No frontend changes are needed; the existing `gameEnded` handler already
     processes `reason: 'win'` correctly.

TASK 2 — Display the player's profile picture in the new-turn banner
--------------------------------------------------------------------------------

CONTEXT:
  When a new turn starts, `game.page.ts` receives a `newTurn` event, reads
  `gameData.gameState.currentTurn` (a MarbleColor), finds the matching player in
  `gameData.gameState.players`, and stores their name and color into Angular
  signals (`newTurnName`, `newTurnColor`) which drive the banner overlay.

  The banner (`game.page.html` lines 58–69) currently shows:
    [marble decoration] | "Nouveau Tour" / player-name | [marble decoration]

  The `Player` type already has an optional `picture?: string` field (a base64
  data URL). The board component already shows player pictures using this field
  (with a color-div fallback when `picture` is absent).

WHAT TO DO:
  Show the player's profile picture inside the new-turn banner, falling back
  gracefully to the existing marble decoration when no picture is available.

  Steps:

  1. `frontend/src/app/game/game.page.ts`
     - Add a new signal: `newTurnPicture = signal<string | null>(null)`.
     - When handling the `newTurn` event, set `newTurnPicture` to
       `player.picture ?? null` alongside the existing name/color signals.
     - Reset `newTurnPicture` to `null` when the banner is dismissed.

  2. `frontend/src/app/game/game.page.html`
     - Replace (or conditionally replace) one of the two `.new-turn-marble`
       decorative elements with the player avatar when a picture is available:
         @if (newTurnPicture()) {
           <img class="new-turn-avatar" [src]="newTurnPicture()" alt="" />
         } @else {
           <div class="new-turn-marble" [class]="'ntm-' + newTurnColor()"></div>
         }
     - Keep the other `.new-turn-marble` on the opposite side for symmetry.

  3. `frontend/src/app/game/game.page.scss`
     - Add styles for `.new-turn-avatar`:
       - Circular crop: `border-radius: 50%`, fixed size matching the marble
         element (check the existing `.new-turn-marble` dimensions in the file).
       - `object-fit: cover` to avoid distortion.
       - A thin colored border using the current turn color (follow the existing
         color-attribute pattern: `[attr.data-color]` is already on the overlay).
       - No extra animation beyond what the parent banner already applies.

  NOTE: Follow the same avatar-with-fallback pattern already used in
  `board.component.html` (lines 20–24 and 154–158) to keep the approach
  consistent across the game UI.
